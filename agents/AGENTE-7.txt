Regras de harmonização:
- Diferencie explicitamente:
  - vocabulário externo da API
  - vocabulário interno de persistência
  - equivalências entre estados internos e estados expostos
- Quando houver diferença aceitável entre camadas, documente o mapeamento.
- Quando houver inconsistência real, corrija diretamente o texto.
- Preserve o caráter didático do projeto.
- Prefira simplicidade e clareza para aprendizado.

Critérios específicos de correção:

A) Estados
- API pública deve usar um vocabulário consistente e explícito.
- Se o modelo interno usar `CAPTURED` e a API usar `SETTLED`, documente o mapeamento.
- Se `CREATED` e `PENDING` coexistirem como estado inicial, remova a ambiguidade explicando a diferença entre estado de borda e estado persistido interno.

B) Modelo externo x interno
- Em `docs/data-state.md`, explicar que `payer` e `payee` são objetos do contrato HTTP e que internamente podem ser normalizados para `customerId` e `merchantId`.
- Em `docs/api/openapi.md`, adicionar observação equivalente.

C) Roadmap
- Se `GET /payments/by-idempotency-key/{idempotencyKey}` já estiver no contrato e no C4, consolidar sua presença na Fase A.
- Na Fase B, deixar apenas evoluções avançadas de retenção, expiração, replay avançado ou comparação canônica de payload.

D) Integration Review
- Remover itens já resolvidos.
- Não manter observações desatualizadas que contradigam os arquivos atuais.
- Transformar o documento em revisão viva e confiável do estado atual.

E) Escopo de idempotência
- Se multi-tenant não estiver formalmente assumido como requisito do MVP, evitar linguagem que imponha `tenantId` como requisito obrigatório.
- Preferir “escopo do cliente autenticado” ou equivalente, com nota de evolução futura para multi-tenant.
