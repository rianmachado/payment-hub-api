# 1) Ajustes obrigatórios (bloqueantes)

1. Remover acoplamento HTTP do `PaymentsService`
   - O `PaymentsService` nao deve “imitar controller”.
   - Ajustar a assinatura/retorno dos metodos para que o service **nao retorne `statusCode`**.
   - O fluxo de criacao deve retornar somente dados do domínio/DTO (ex.: `CreatePaymentResponseDto`) e um sinal interno (ex.: `idempotencyReplay: true`) para o controller decidir se retorna `201` ou `200`.
   - Garantir que a responsabilidade de “resposta HTTP completa” permaneça para o controller (commit futuro).

2. Encaixe mínimo com `Transactions` e `Providers` (stub/injeção)
   - Mesmo sem implementar a integração final (PSP real), o `PaymentsService` precisa passar por uma etapa mínima de orquestração que encadeie o resto do domínio:
     - injeção de abstrações/stubs mínimos de `TransactionsService` e `ProvidersService` (ou equivalente).
     - no caminho `first_call`, executar uma chamada stub coerente com a orquestração (por exemplo: criar/preparar uma transacao e preparar a mudança inicial de estado suportada pelo baseline).
     - no caminho `replay idempotente`, garantir que o PSP/provedor nao seja acionado (e, portanto, o stub de providers nao deve ser chamado no replay).
   - Nao deixe o fluxo apenas com `TODO` sem um “hook” mínimo de integração.

----------------------------------------
# 2) Ajustes recomendados (não bloqueantes)

1. Fail-fast para invariantes do DTO
   - Se `payer`/`payee` nao contiverem `id` nem `externalId`, evitar “silenciosamente” persistir identificadores vazios.
   - Fazer o `PaymentsService` lançar excecao/erro consistente (código `PAYMENT_VALIDATION_ERROR` ou equivalente), evitando criar/consultar pagamentos inválidos por dados incompletos.

2. Padronizar mensagens/textos de erro
   - Manter consistência textual (incluindo acentos e tom) para reduzir divergências no tratamento/observabilidade.
